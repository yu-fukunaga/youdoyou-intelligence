# ブランチ作成

main から新しいブランチを作成し、切り替えるまでの一連の操作を行う。

このプロジェクトにはチケット管理システムがないため、ブランチ名にチケット番号は含めない。

## ワークフロー

1. **現状確認**
   - `git status` で現在のブランチ・未コミット変更の有無を確認する
   - ユーザーから未コミット変更の扱いについて指示があればそれに従う
   - 指示がなければ、未コミット変更は新しいブランチにそのまま引き継ぐものとして扱う(stashなどは行わない)

2. **main の最新化**
   - `git fetch origin main` で origin/main を最新化する

3. **ブランチ名の決定**
   - 会話の文脈から作業内容を把握し、`{prefix}/{kebab-case-summary}` の形式で名前を決める
   - 会話に作業内容の手がかりがない場合は、`git status` / `git diff` (必要なら `git diff --cached` も) で現在の変更内容を確認し、そこから内容を推測する
     - 変更内容が単一の目的にまとまっている場合はそのまま推測して進める
     - 変更内容に複数の異なる目的が混在している場合(例: 無関係な機能追加とバグ修正が同時に含まれる等)は、自動で決めつけず、どの変更をこのブランチに含めるか・ブランチ名をどうするかをユーザーに確認する
   - prefix は以下から選ぶ([commit skill](../commit) のgitmoji分類に準拠):
     - `feature` — 新機能の追加
     - `fix` — バグ修正
     - `refactor` — リファクタリング(機能変更なし)
     - `chore` — テスト・CI・設定変更・依存関係更新などの雑務
     - `docs` — ドキュメントの更新
   - **重要**: `scripts/create_release_tag.sh` は `feature/*` ブランチ由来のマージかどうかでリリースのminor/patchバージョンアップを判定している。`feature` を選ぶとminorバンプ、それ以外はpatchバンプになるため、新機能かどうかの判断を誤らないこと
   - `{prefix}/{summary}` のブランチ名候補を3種類作成し、ユーザーに提示して選んでもらう
     - 候補はprefixまたはsummaryの表現を変えるなどして、内容が異なる観点から区別できるようにする
     - ユーザーが選んだ候補、またはユーザー自身が挙げた別名を採用してから次に進む

4. **ブランチ作成・切り替え**
   - `git checkout -b {prefix}/{summary} origin/main` を実行する
   - 未コミット変更がある場合、git がこの操作で自動的に新ブランチへ引き継ぐ
   - コンフリクトなどで失敗した場合は自動解決せず、状況をユーザーに報告して止める

5. **結果報告**
   - `git status` で結果を確認し、切り替わったブランチ名と作業状態を簡潔に報告する
